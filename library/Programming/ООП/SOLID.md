# SOLID
## Определение
SOLID - набор из 5 принципов объектно ориентированного проектированния программы. Созданы для написания поддерживаемого и расширяемого кода.
## SRP -Single Responsibility Principle
Принцип единственной ответственности - каждый класс (или модуль) должен иметь только одну причину для изменения, то есть отвечать только за одну зону ответственности. Если класс имеет несколько обязанностей, то следует разбить его на классы поменьше, каждый из которых будет решать свою задачу. Но также чрезмерное разбиение класса на классы поменьше может усложнить архитектуру и стать антипаттерном, поэтому при разбиении нужно руководствоваться здравым смыслом и пониманием того, чего достаточно для программы.

Плохой пример
```cs
public class Player
{
	private int _health;
	private int _gold;
	private int _damage;
	private float _speed;

	private void Attack()
	{
		. . .
	}

	private void Move()
	{
		. . .
	}

	private void PickGold()
	{
		. . .
	}
}
```
Хороший пример
```cs
public class Player
{
	private int _health;

	// Каждый класс выполняет свою задачу
	private Weapon _weapon; // атака
	private Movement _movement; // передвижение
	private Wallet _wallet; // деньги
}
```
## OCP - Open/Closed Principle
Принцип открытости для расширения, закрытости для изменения:
 - открытость для расширения: поведение сущности может быть расширено путём создания новых типов сущностей (производных классов)
- закрытость для изменения: в результате расширения поведения сущности, не должны вноситься изменения в код, который эту сущность использует. Тоесть необходимо изначально предусмотреть структуру объекта, для того, чтобы не менять его в будущем.
## LSP - Liskov Substitution Principle
Принцип подстановок Барбары Лисков - объекты базового класса должны быть заменяемы объектами производных классов без изменения корректности программы. Если класс `B` наследуется от класса `A`, то мы должны быть уверены, что сможем везде использовать `B` вместо `A` без поломки логики программы.
## ISP - Interface Segregation Principle
Принцип разделения интерфейсов - програмные сущности не должны зависеть от методов, которые они не используют. Вместо определения одного большого интерфейса нужно определить несколько маленьких интерфейсов, каждый из которых будет выполнять свою логическую роль.
## DIP - Dependency Invertion Principle
Принцип инверсии зависимостей - модули верхнего уровня не должны зависеть от модулей нижнего уровня, оба зависят от абстракций. Абстракции не должны зависеть от деталей.
- Абстракция - обобщение семейсива объектов, выделение общей сути.
- Классы низкого уровня - конкретные объекты, имеющие общее семейство, но разные детали.
- Классы высокого уровня - классы обрабатывающие группы классов низкого уровня.
- Детали - уникальные элементы классов низкого уровня, присущие только этим классам.
Архитектура кода должна быть основана на абстракции, а не на конкретных реализациях.

Определение для экзамена:
Программа состоит из слоев, каждый слой состоит из модулей. Каждый модуль может обращаться либо к модулю из своего слоя, либо к модулю из слоя ниже.

Пример для экзамена:
Слои: интерфейс, сервисы, модели, база данных.
Интерфейс обращается к сервису. Сервис может обратиться только к другим сервисам либо к моделям. Модели также могут обращаться только к моделям или к базе данных.
